Executive Summary

What ORBIS is and why it exists

ORBIS is a platform-agnostic operator training and evaluation system that decouples scenario design, physics simulation, sensor modelling, and after-action analysis into independently replaceable components connected by open XML data formats.

At its core, ORBIS consists of three products: an Instructor Operator Station (IOS) for designing and injecting training scenarios; a real-time telemetry pipeline that captures operator and platform state at configurable rates; and a C++20 range-based query engine that evaluates performance against scenario criteria and generates structured after-action reports.

Unlike conventional training simulators — where evaluation logic is hard-coded for one aircraft or vessel type — ORBIS exposes a clean PlatformModel interface. Swapping the helicopter backend for a naval vessel or UAV requires implementing one interface. The IOS, query engine, and report layer require zero modification.

"The flight model is a replaceable component. ORBIS is the intelligence layer above it — the system that knows what a good operator looks like, and can prove it with data."

02

The Problem

Why existing training simulators fall short

Defense and naval training simulators share a fundamental architectural problem: evaluation logic, scenario control, and telemetry analysis are tightly coupled to a single platform. When a contractor builds a helicopter trainer, they build a custom IOS, a custom data format, and custom evaluation criteria. When they build a naval trainer next, they start from scratch.

Three consequences emerge from this coupling:

Problem Current state ORBIS solution
Platform lock-in Evaluation logic re-written for each new platform type PlatformModel interface — swap backends, keep everything else
Rigid evaluation Pass/fail criteria hard-coded at build time by software engineers Composable query engine — instructors define criteria at runtime
Proprietary data Telemetry in vendor-specific binary formats, not interoperable Open XML schemas — importable into any analysis tool
Batch-only analysis After-action review only available post-mission Lazy range views — live evaluation during the mission

Product Vision

What ORBIS looks like as a deployed system

An instructor sits at the ORBIS IOS. She selects a helicopter scenario — Mountain NOE, degraded visibility, partial engine failure at T+120s. She sets wind, cloud ceiling, and the waypoint altitude band she expects the trainee to hold. She clicks Deploy Scenario.

ORBIS writes the scenario XML, initialises the JSBSim Bo 105 flight dynamics model, and begins streaming telemetry at 10Hz. On her second monitor, live metrics update: altitude deviation at each waypoint, rotor RPM trend, reaction time to the engine failure. The query engine runs continuously against the live ring buffer.

At mission end, ORBIS generates a structured after-action report: waypoint compliance, altitude exceedance events with timestamps, engine response assessment. The report is exportable as XML for integration into a learning management system, or rendered as a formatted HTML document for the debrief.

Next week the same instructor runs a naval gunnery scenario on a different vessel model. The IOS looks identical. The query engine asks the same structural questions — was the operator within criteria, when did they deviate, how quickly did they recover. Only the platform backend and scenario parameters change.

Core Components

The universal data contract and abstract platform interface

TelemetrySample — Universal Data Contract

include/orbis/TelemetrySample.hpp C++20 
struct TelemetrySample {
    // ── Universal fields — every platform ──────────────────
    double      timestamp_sec;
    double      latitude_deg;
    double      longitude_deg;
    double      altitude_m;
    double      heading_deg;
    double      speed_ms;
    double      pitch_deg;
    double      roll_deg;
    double      yaw_rate_dps;

    // ── Platform-specific — optional ───────────────────────
    std::optional<double> rotor_rpm;         // helicopter
    std::optional<double> engine_torque_pct;  // helicopter
    std::optional<double> fuel_flow_gph;      // helicopter
    std::optional<double> sea_state_douglas;  // naval
    std::optional<double> rudder_angle_deg;   // naval
    std::optional<double> gun_elevation_deg;  // naval gunnery

    // ── Synthesised sensor outputs ─────────────────────────
    std::optional<double> radar_alt_ft;
    std::optional<double> eoir_target_range_m;
    std::optional<bool>   eoir_target_locked;
    std::optional<double> surface_contact_range_m;

    // ── Evaluation context ─────────────────────────────────
    std::string              flight_phase;
    bool                     waypoint_reached  = false;
    std::vector<std::string> active_failures;
};

PlatformModel — Abstract Interface

include/orbis/PlatformModel.hpp C++20 
class PlatformModel {
public:
    virtual ~PlatformModel() = default;

    virtual void            initialise(const ScenarioParams&) = 0;
    virtual void            step(double dt_sec)              = 0;
    virtual TelemetrySample read()                          = 0;
    virtual void            inject(const FailureEvent&)    = 0;
    virtual std::string     platform_type()            const = 0;

    // Factory — reads platform type from scenario XML
    static std::unique_ptr<PlatformModel>
    create(const std::string& type, const ScenarioParams&);
};

Query Engine

Composable lazy evaluation over telemetry streams using C++20 ranges

The ORBIS query engine is built on std::ranges and lazy view composition. Telemetry is held in a std::vector<TelemetrySample> or a live ring buffer. Evaluation queries are composable pipelines that run without copying or materialising intermediate data.

Because views are lazy, the engine operates identically on a live mission ring buffer — giving the instructor real-time evaluation metrics — and on a post-mission XML log for detailed after-action analysis.

src/orbis/QueryEngine.cpp — example evaluation queries C++20 ranges 
// ── Q1: Altitude compliance at waypoint ─────────────────
auto at_alpha = telemetry
    | std::views::filter([](auto& s) {
        return geo_distance(s, waypoint_alpha) < 100.0;
      })
    | std::views::filter([](auto& s) {
        return s.altitude_m >= 60.0 && s.altitude_m <= 150.0;
      });
bool compliant = !std::ranges::empty(at_alpha);

// ── Q2: Cluster low-altitude exceedance events ──────────
auto exceedances = telemetry
    | std::views::filter([](auto& s) { return s.altitude_m < 30.0; })
    | std::views::chunk_by([](auto& a, auto& b) {
        return b.timestamp_sec - a.timestamp_sec < 2.0;
      });

// ── Q3: Engine torque trend during approach phase ────────
auto approach_torque = telemetry
    | std::views::filter([](auto& s) {
        return s.flight_phase == "approach";
      })
    | std::views::transform([](auto& s) {
        return s.engine_torque_pct.value_or(0.0);
      });

// ── Q4: Naval — sustained contacts within 5 nm ──────────
auto threats = telemetry
    | std::views::filter([](auto& s) {
        return s.surface_contact_range_m
                   .transform([](auto r) { return r < 9260.0; })
                   .value_or(false);
      })
    | std::views::chunk_by(time_contiguous(30.0));

Key Differentiators

What makes ORBIS technically distinct
01
Platform agnostic by design
One PlatformModel interface. Helicopter, naval, UAV, and ground vehicle backends are interchangeable. IOS and query engine require zero modification between platforms.
02
Live + batch evaluation
C++20 lazy range views operate identically on a live ring buffer during a mission and on a post-mission telemetry log. Instructors see real-time performance metrics without batch overhead.
03
Open data formats
Scenario and telemetry XML schemas are fully open and documented. Data is importable into Python, MATLAB, or any learning management system without vendor-specific tooling.
04
Industry-grade flight model
JSBSim flight dynamics — the same engine used by NASA, the USAF, and as the physics core of FlightGear. Not a toy model. Aerodynamic coefficients from real aircraft data.
05
Composable sensor synthesis
Radar altimeter, EO/IR boresight, and surface search radar are modelled as separate synthesised outputs on top of the flight dynamics — the same architectural separation HENSOLDT uses in Simsphere.
06
ROS2 compatible
Optional ROS2 publisher outputs telemetry as nav_msgs/Odometry topics. ORBIS scenarios can drive robotics simulation environments, making the platform directly relevant to autonomous systems validation.

Development Roadmap

12-weekend build plan from core slice to multi-platform demo
Milestone 1 — Core slice Weekends 1–2
JSBSim Bo 105 integration. PID autopilot flying a 3-waypoint route. TelemetrySample struct. XML writer. One range-view query — altitude compliance at waypoints. Plain text report output. End-to-end pipeline proven.
Milestone 2 — Sensor synthesis Weekends 3–4
Radar altimeter model with Gaussian noise. EO/IR boresight geometry and target detection logic. Engine failure injection via FailureEvent interface. Sensor outputs written to telemetry XML.
Milestone 3 — IOS application Weekends 5–6
Qt desktop IOS UI. Scenario designer panels: initial conditions, environment, waypoints, failure injection. Scenario XML generation. Live telemetry monitor with real-time parameter display.
Milestone 4 — Full query engine Weekends 7–8
Complete range-view query library. Phase detection, deviation event clustering, waypoint compliance scoring, engine response assessment. HTML after-action report generation. Live ring buffer evaluation mode.
Milestone 5 — Naval backend Weekends 9–10
NavalVesselModel implementing PlatformModel interface. Ship kinematics with sea state effects. Surface search radar contact synthesis. Naval scenario XML schema. Contact tracking query examples.
Milestone 6 — Demo & integrations Weekends 11–12
